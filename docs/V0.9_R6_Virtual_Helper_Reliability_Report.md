# v0.9 R6 Virtual Helper Reliability Report

Version: `0.9.0-dev`
Status vocabulary: DONE / PARTIALLY DONE / BLOCKED / NOT STARTED / NOT VERIFIED

## Root Cause

The parent persisted global Journal stage `virtualDisplayStarted` after observing only the helper process. The child could consume its exact-stage capability later, after the parent had changed `keepAwakeStarted` to `virtualDisplayStarted`, and correctly reject authorization.

## Launch Ordering

Before:

```text
reserve capability at keepAwakeStarted
-> launch process
-> observe PID
-> write global virtualDisplayStarted
-> child consumes capability (racy)
-> create display
```

After:

```text
persist intent
-> reserve exact-stage capability
-> launch process and persist resource started
-> child consumes capability at the unchanged global stage
-> child emits CH_HELPER_AUTHORIZED and waits
-> parent validates IDs and persists provisional ownership
-> parent sends exact CH_PARENT_CONTINUE
-> child creates CGVirtualDisplay and emits CH_VIRTUAL_DISPLAY_READY
-> parent accepts a running host plus display ID
-> persist resource observed and global virtualDisplayStarted
-> commit RuntimeState
```

If the parent exits before continuation, stdin closes and the child creates no display. A process-only state remains recoverable at resource stage `started`; it is never described globally as `virtualDisplayStarted`.

## Compensation Ownership

- A resource manager cleans only its own uncommitted helper/process.
- The outer Enable catch owns workflow-wide prepared-resource compensation.
- Handoff recovery marks compensation `inProgressOrPaused` before it starts, so the outer catch cannot repeat it.
- The original virtual-helper error is retained as the cause; cleanup diagnostics are separate.

# v0.9 R7.2 Audit Remediation Report

Development version: `0.9.0-dev`
Date: 2026-07-12

## R7.2-P0-01 Atomic Uninstall: DONE

R7.1 performed a strict uninstall check while holding `WorkflowOperationLock`, released that lease, and then let the shell remove the installed App and CLI. Another Enable or configuration mutation could therefore begin between verification and deletion.

R7.2 selects a copied coordinator architecture. The shell first requires successful Restore, creates a private `0700` temporary directory, copies a trusted CLI, and verifies byte identity with SHA-256. That copied executable invokes `uninstall-session`. Core validates exact production paths or confined test paths, acquires `WorkflowOperationLock`, reruns strict safety checks while the lease is held, unloads and verifies the LaunchAgent, quits and verifies the exact App executable, then removes LaunchAgent, App and standalone CLI in that order. The lease is released only after deletion completion or a returned failure.

If preflight, entry shutdown, LaunchAgent deletion or App deletion fails, the installed CLI is preserved. If CLI deletion fails, the failure names its exact remaining path. Configuration, logs, RuntimeState and Recovery Journal are never uninstall targets. A deterministic cross-process test proves a second process cannot acquire the lock at the deletion barrier.

## R7.2-P1-01 Operational Evidence: DONE

`OperationalEvidence` is independent from `CleanNormalAssessment` and contains typed RuntimeState read status, Journal evidence, process snapshot outcome, managed owner observations, display IDs and violations. The assessor validates phase-dependent Preparing ownership, Confirm Required replacement/rollback coherence, Headless ownership and built-in handling, Journal compatibility and exact managed display identity.

The App maintains a separate lock-protected cache. Refresh retains the previous completed evidence, stale evidence becomes `Unable to verify`, and refresh failure preserves history without reporting false health. Non-Normal menus consume only this cache. Normal menus continue to use Clean Normal. Periodic reconcile captures one process snapshot and one display enumeration, reuses both for manager observations, and runs off the main thread. Lock contention fails the refresh and preserves prior evidence rather than launching an unlocked duplicate scan.

Enable and Confirm completion route to Operational Evidence whenever final mode is non-Normal. Successful Restore clears Operational Evidence and refreshes Clean Normal. Status and Doctor perform explicit typed assessments. Transition diagnostics compare observed facts rather than timestamp/source/duration, so unchanged cycles are deduplicated.

## Independent Audit

The final audit re-read current source rather than relying on R7.1/R7.2 implementation claims. It identified and fixed four incomplete consistency paths before the final gate: Preparing phase ownership, Confirm Required replacement/rollback state, Headless built-in enumeration agreement, and explicit Error presentation precedence. It also removed duplicate scans on workflow-lock contention and corrected diagnostic timestamp noise.

No unresolved source-level P0 or P1 was found after those fixes. R7.2 contains no separate P2 item, and no material P2 was identified in the final scope audit.

## Remaining Risks

- Private display and Touch Bar APIs remain hardware/OS-sensitive and require product-owner UAT.
- Killing the uninstall coordinator after App deletion can leave the standalone CLI installed by design; this preserves recovery rather than claiming atomic filesystem rollback.
- Remote CI execution and real Intel/Apple Silicon hardware cycles were not performed in this task.

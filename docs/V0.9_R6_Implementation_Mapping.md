# v0.9 R6 Implementation Mapping

Status: active implementation evidence. Historical reports are not treated as proof.

## System Model

- `RuntimeState` is a presentation/commit record, not independent resource proof.
- `RecoveryJournal` is the durable workflow authority and must remain exact-stage compatible with one-time helper capabilities.
- `WorkflowOperationLock` serializes App/CLI lifecycle work; Clean Normal is reassessed after lock acquisition.
- Keep Awake and virtual-display resources use intent -> process start -> ownership observation -> RuntimeState commit, retaining recovery evidence until independently verified absent.
- A managed virtual replacement is safe for handoff only after helper authorization, a ready display ID, and ownership evidence are accepted.
- The outer Enable workflow owns workflow-wide compensation; a manager may only clean its own uncommitted resource.
- Internal helpers are separate processes. They use stdout for protocol events and stderr for diagnostics; they must not own the parent file log.

## Issue Map

### R6-P0-01 - Capability and Journal-stage race

- Current code: `VirtualDisplayManager.createVirtualDisplay`, `HelperCapabilityStore`, `CLIInternalHelperCommands`.
- Trigger: parent captures helper process facts and advances Journal before the child consumes a capability reserved at `keepAwakeStarted`.
- Error path: child rejects exact-stage authorization after Journal becomes `virtualDisplayStarted`.
- Safety invariant: reserve at S -> helper consumes at S -> parent may advance beyond S.
- Planned files: `VirtualDisplayManager.swift`, `HelperCapability.swift`, `CLIInternalHelperCommands.swift`, protocol parser/launcher support.
- Planned tests: delayed authorization barrier, wrong identifiers, duplicate/partial events, one-time consumption.
- Acceptance commands: native build/test, architecture builds, critical-read and responsiveness audits, self-test.

### R6-P0-02 - Truthful virtual helper launch state

- Current code: `VirtualDisplayManager.createVirtualDisplay` writes global `virtualDisplayStarted` after process ownership only.
- Trigger: helper launched but authorization/display creation has not completed.
- Error path: Journal claims a globally started display while only a process exists.
- Safety invariant: global `virtualDisplayStarted` requires accepted display-ready evidence; provisional ownership remains recoverable at resource stage `started`.
- Planned files: `VirtualDisplayManager.swift`, `Models.swift` only if existing substages are insufficient.
- Planned tests: process-only, authorized-before-ready, ready-and-running, ready-then-exit, persistence failure windows.
- Acceptance commands: lifecycle tests plus source audit that no premature global-stage write remains.

### R6-P1-01 - Immediate helper early-exit detection

- Acceptance commands: deterministic elapsed-time assertions and native tests.
